# FABLE5 互動設計

## 兩階段減法

- 第一階段(v4.0):把 50 頁 SOP 拆成 CORE + scripts + docs
- 第二階段(v4.1):讓使用者不必閱讀那些文件

SOP 應該存在於系統行為裡,而不是要求使用者先讀懂整份文件。
CORE_WORKFLOW.md 是維護與稽核文件;`scripts/fable.sh` 與 `/battle`
才是使用者接觸的流程入口。

## 輸出契約:每步只回答三件事

1. 下一步要做什麼
2. 需要提供什麼資訊(附可複製的範例)
3. 完成後會發生什麼

## 判斷標準

> 能由命令查出的,不問人;能由程式執行的,不叫人手動做。

系統自動判斷(不詢問):Git 乾淨與否、bisect session、Port 占用、
必要工具、腳本語法、證據包完整性、JSON 合法性、禁區違規、
worktree 是否清乾淨。

只詢問系統查不到的:START_COMMAND、任務描述、good commit、
人類裁決確認。

## 精準阻擋

發現問題時停在當前步驟,錯誤訊息固定五要素:

```text
狀態(哪個階段、哪一項 ✗)
原因
需要處理(具體、可複製)
重試方式(修好後執行哪個命令)
未執行事項(系統因此沒做什麼——保證錯誤不向後傳播)
```

FABLE5 永不自動終止未知程序、永不覆蓋既有證據。

## 漸進揭露

預設輸出保持簡單;`--verbose` 顯示實際命令、路徑、exit code。
一般使用者不需要知道 exit 125;維護者遇到問題時仍能看到全部細節。

## 責任分工

| 誰 | 做什麼 |
|---|---|
| fable.sh | 機械步驟:前置檢查、建目錄/worktree、跑測試、驗證據、清理、續跑判斷 |
| /battle(Claude Code) | LLM 步驟:提案、交叉審查、實作、裁決;機械步驟一律委派 fable.sh |
| 人類 | 系統查不到的輸入 + 裁決確認 + patch 合入 + 封存前敏感檢查 |
| CORE_WORKFLOW.md | 定義契約與驗收,供維護者查閱與稽核 |

## 完成標準(盲測)

找一位沒讀過 CORE 的人,只告訴他執行 `bash scripts/fable.sh`。
如果他能:完成前置設定、知道錯誤怎麼修、完成一次沙盒 Battle、
找到最終結果、全程不需要打開 CORE——第二階段減法才算通過。
